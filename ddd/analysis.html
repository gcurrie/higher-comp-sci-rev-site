Database Analysis

DDD Unit 2 Topic 1 of 7

The analysis stage involves understanding what data a database needs to store, how that data is organised, and how different pieces of data relate to each other.

Key Concepts

Entity
A real-world object or concept about which data is stored. Becomes a table in the database. Examples: Pupil, Course, Teacher, Product.
Attribute
A piece of information stored about an entity. Becomes a field (column) in the table. Example: a Pupil entity might have attributes: PupilID, Name, DateOfBirth, YearGroup.
Relationship
A link between two entities showing how they are connected. Example: a Pupil is enrolled on a Course.
Primary Key
A unique identifier for each record in a table. No two records can share the same primary key value. Example: PupilID.
Foreign Key
A field in one table that refers to the primary key of another table, creating a link between them.

Types of Relationship

Relationships between entities describe how many instances of one entity can be associated with instances of another:

Relationship TypeMeaningExample
One-to-One (1:1)One record in table A links to exactly one record in table BA person has one passport
One-to-Many (1:M)One record in table A links to many records in table BOne teacher teaches many pupils
Many-to-Many (M:M)Many records in table A link to many records in table BMany pupils enrol on many courses
ℹ️
Many-to-many relationships cannot be directly implemented in a relational database. They must be resolved by creating a linking table (also called a junction or associative table) which breaks the M:M into two 1:M relationships.

Worked Example — School Database

Consider a school that wants to store information about pupils and the clubs they attend.

Identifying entities

  • Pupil — stores information about each pupil
  • Club — stores information about each club

Identifying attributes

  • Pupil: PupilID (PK), FirstName, LastName, YearGroup, DateOfBirth
  • Club: ClubID (PK), ClubName, MeetingDay, TeacherInCharge

Identifying relationships

A pupil can join many clubs, and a club can have many pupils — this is a many-to-many relationship. It requires a linking table:

  • PupilClub: PupilID (FK), ClubID (FK), DateJoined

Practice Questions

Question 1 — 2 marks

Explain the difference between an entity and an attribute. Give an example of each from a library database.

Question 2 — 1 mark

State what is meant by a primary key.

Question 3 — 2 marks

A hospital stores data about patients and doctors. Describe the relationship between these two entities and explain whether it is one-to-one, one-to-many, or many-to-many.

Question 4 — 3 marks

A cinema database stores films and customers. A customer can book tickets for many films, and a film can be booked by many customers. Explain why a linking table is needed and suggest appropriate fields for it.
