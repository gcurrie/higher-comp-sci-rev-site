Tables and Keys

Each entity in an ER diagram becomes a table in the relational database. Relationships are implemented using keys.

TermDefinition
Primary Key (PK) An attribute (or combination of attributes) that uniquely identifies every record in a table. Cannot be NULL; must be unique.
Foreign Key (FK) An attribute in one table that refers to the primary key of another table. Used to create the link between related tables.
Compound Key A primary key made of two or more attributes that together are unique (used in linking tables).
💡
In a one-to-many relationship, the foreign key always goes in the table on the "many" side. For example, if one Teacher teaches many Classes, the ClassTable gets a TeacherID foreign key.

Data Dictionary

A data dictionary is a formal document that describes every attribute in the database in detail. It is produced during the design stage, before the database is built.

For each attribute, the data dictionary records:

  • Entity Name — which table the attribute belongs to
  • Attribute Name — the name of the field/column
  • Key — whether it is a Primary Key (PK), Foreign Key (FK), or neither
  • Data Type — the type of data (integer, text, date, etc.)
  • Size — the maximum length or number of digits
  • Validation — rules that constrain what data can be entered

Validation Types

Validation ensures only sensible, correctly formatted data can be entered into the database.

Validation TypeWhat it checksExample
Presence check The field must not be left empty — a value must be entered CustomerName cannot be NULL
Range check The value must fall within a specified minimum–maximum range Mark must be between 0 and 100
Type check The value must be of the correct data type Age must be an integer, not text
Restricted choice Only specific permitted values are allowed Gender must be "M", "F" or "Other"
Length check The value must be (or not exceed) a specified number of characters Postcode must be between 6 and 8 characters
