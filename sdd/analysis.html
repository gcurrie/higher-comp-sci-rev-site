Purpose

The purpose statement describes, in plain language, what the software will do and for whom. It answers the question: "Why does this system need to exist?"

A good purpose statement is:

  • Clear and unambiguous
  • Written for a non-technical audience
  • Short — usually one or two sentences
Example Purpose Statement
"A stock management system for a small bookshop that allows staff to search for book availability, add new stock and generate low-stock alerts."

Scope & Boundaries

The scope defines exactly what the system will and will not do. Establishing clear boundaries at the analysis stage prevents scope creep — where a project grows beyond what was originally planned.

In scope (will do)
  • Search stock by title or author
  • Add, edit and delete stock records
  • Display items below minimum stock level
Out of scope (will not do)
  • Process customer payments
  • Manage staff accounts or permissions
  • Order books from suppliers automatically

When answering exam questions about scope, always identify what is included and what is deliberately excluded.

Functional Requirements

Functional requirements describe what the system must do. They are broken into three categories:

CategoryDefinitionBookshop example
Inputs Data entered into the system Book title, author name, ISBN, quantity in stock
Processes Actions performed on data Search for a book, calculate stock totals, flag low stock
Outputs Information produced by the system Search results list, low-stock report, stock quantity displayed
💡
Exam questions often ask you to identify the inputs, processes and outputs from a description. Look for data going in (inputs), calculations or decisions (processes) and results shown to the user (outputs).

End-user Requirements

End-user requirements capture what the people who will use the software need from it. These are distinct from technical requirements and focus on usability, accessibility and user experience.

Examples of end-user requirements:

  • The interface must be operable by staff with no IT background
  • Search results must appear in under two seconds
  • Text must be readable on standard monitors without zooming

End-user requirements feed directly into the usability evaluation criterion — during evaluation, you assess whether the software meets these agreed requirements.

Worked Example

The following shows a complete analysis section for a typical Higher-level programming task.

Scenario: School Exam Results System

"Create a program that stores pupil exam results, finds the highest and lowest marks, and counts how many pupils achieved grade A (70% and above)."

Purpose: A program to store and analyse exam results for a class of pupils, allowing a teacher to quickly identify top performers and count grade A passes.

Scope: The program will store names and marks; calculate highest, lowest and count of grade As. It will not handle multiple subjects, calculate averages, or store the data permanently.

Functional Requirements:

CategoryDetail
InputsPupil names (string), exam marks (integer 0–100), class size (integer)
ProcessesStore names and marks; find highest mark; find lowest mark; count marks ≥ 70
OutputsHighest mark (and pupil name); lowest mark (and pupil name); count of grade A passes
