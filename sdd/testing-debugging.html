Test Plans

A comprehensive test plan is created before testing begins. It defines exactly what will be tested, what data will be used, what the expected result is, and what the actual result turned out to be.

Types of Error

⚠️
The SQA Higher specification uses the term "execution error" — NOT "runtime error". Use the correct SQA terminology in your exam answers.
Error TypeWhen DetectedDescriptionExample
Syntax error During translation (before running) Code that violates the grammar rules of the language WHLIE i < 10 DO (misspelled keyword)
Execution error At runtime (while the program is running) An operation that cannot be carried out — causes the program to crash Dividing by zero; accessing an array index that doesn't exist
Logic error During testing (program runs but gives wrong output) The code runs without crashing but produces incorrect results Using > instead of >= in a condition; adding when you meant to subtract

Categories of Test Data

Three categories of test data are used to thoroughly test a program:

Normal

Typical valid inputs that the program is designed to handle. Should produce the expected output.

Example: mark of 65 → expects Grade B
Extreme (Boundary)

Valid values at the very edge of the accepted range. These are the minimum and maximum allowed values.

Example: marks of 0 and 100 (if 0–100 is valid range)
Exceptional

Invalid inputs that the program should not accept. Used to test that error handling works correctly.

Example: entering "hello" for a numeric field; -5 for a mark
💡
In the exam you may be asked to choose appropriate test data and justify why each piece falls into its category. For each value, explain what it tests — don't just list numbers without explanation.

Dry Runs and Trace Tables

A dry run is the process of manually tracing through a piece of code step by step, recording the value of variables at each point. A trace table is the table used to record these changing values.

Trace Table Example — Find Maximum
marks = [45, 78, 31, 92, 60]
maxIndex = 0

i=1: marks[1]=78 > marks[0]=45 → maxIndex = 1
i=2: marks[2]=31 > marks[1]=78? NO
i=3: marks[3]=92 > marks[1]=78? YES → maxIndex = 3
i=4: marks[4]=60 > marks[3]=92? NO

Debugging Tools

Modern IDEs provide debugging tools that allow programmers to pause and inspect program execution without manually modifying code.

ToolWhat it doesUsed for
Breakpoint Marks a specific line in the code where execution will pause Checking the state of all variables at a known point in the program
Watchpoint Pauses execution whenever a specific variable changes to a particular value Tracking down exactly where a variable takes on an unexpected value
ℹ️

Key difference:

  • A breakpoint stops at a line of code
  • A watchpoint stops when a variable's value changes to something specific

Example Scenario

Suppose your find-maximum algorithm is returning an incorrect index. You might:

  1. Set a breakpoint at the start of the FOR loop to check the initial state of maxIndex
  2. Set a watchpoint on maxIndex to pause every time it changes — so you can see which comparison caused it to update
  3. Step through the loop line by line, checking that maxIndex updates correctly each time a larger value is found
