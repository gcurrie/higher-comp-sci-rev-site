The Five Evaluation Criteria

⚠️
The SQA Higher SDD evaluation criteria are: fitness for purpose, efficient use of coding constructs, usability, maintainability, and robustness. "Readability" is a National 5 criterion — do not use it at Higher.
CriterionKey question
Fitness for purposeDoes the software do what it was designed to do?
Efficient use of coding constructsIs the code written as efficiently as possible — loops used instead of repetition, subprograms used for reuse?
UsabilityIs the software easy for the intended user to use?
MaintainabilityCan the code easily be understood and updated by another programmer?
RobustnessDoes the program handle unexpected or invalid input without crashing?

1. Fitness for Purpose

A solution is fit for purpose if it meets all the requirements agreed during the analysis stage. This is assessed against the functional requirements (inputs, processes and outputs) defined at the start of the project.

How to evaluate it

  • Compare the program's outputs against the expected outputs defined in the requirements
  • Run a comprehensive test plan and check all tests pass
  • Confirm all functional requirements are implemented
Example Evaluation Comment

"The program correctly accepts pupil names and marks, stores them in an array of records, displays the highest and lowest marks, and counts grade A passes. All test cases in the test plan passed. The solution is fit for purpose."

2. Efficient Use of Coding Constructs

Efficient code uses the available constructs (loops, conditions, subprograms, arrays) in the most appropriate way — avoiding unnecessary repetition, reducing code complexity, and making good use of modularisation.

Signs of efficient code

  • Loops used instead of copy-pasting similar code blocks
  • Functions and procedures used for operations performed more than once
  • Arrays used to store collections of related values, rather than many separate variables
  • Conditions simplified — e.g. IF flag = TRUE rather than IF flag = TRUE THEN SET result TO TRUE

Signs of inefficient code

  • Copy-pasted blocks of near-identical code that could be in a loop
  • Separate variables (name1, name2, name3…) where an array should be used
  • Repeated calculations not stored in a variable

3. Usability

Usability is how easy the software is for its intended users to use effectively. It is assessed against the end-user requirements defined during analysis.

Factors affecting usability

  • Clear, unambiguous prompts that tell the user exactly what to enter
  • Meaningful error messages that explain what went wrong and how to correct it
  • Logical screen layout and navigation flow
  • Appropriate use of menus, buttons and labelling
  • Consistent interface design throughout the program
Example Evaluation Comment

"The program displays clear prompts at each input stage and shows an error message when the user enters a mark outside 0–100. The menu options are labelled plainly and the interface matches the wireframes agreed at design. Usability is good — a non-technical user can operate the program without assistance."

4. Maintainability

A program is maintainable if another programmer can understand the code and make changes to it with minimum difficulty. This is important because software often needs to be updated after it is first delivered.

What makes code maintainable

  • Meaningful variable names — totalMark is clearer than x
  • Modular structure — separate procedures and functions for distinct tasks
  • Internal commentary — comments explaining the purpose of non-obvious sections
  • Consistent indentation — makes the structure of loops and conditions easy to read
  • No magic numbers — use named constants rather than unexplained literal values

5. Robustness

A robust program handles unexpected or invalid input gracefully — it does not crash, produce nonsensical output, or enter an infinite loop when the user does something unexpected.

Techniques for robustness

  • Input validation loops that reject out-of-range values and re-prompt the user
  • Checking for empty inputs or incorrect data types
  • Checking an array is not empty before attempting to find min/max
  • Providing informative error messages rather than crashing silently
Input Validation — Robust Mark Entry
RECEIVE mark FROM KEYBOARD
WHILE mark < 0 OR mark > 100 DO
    SEND "Invalid — enter a mark between 0 and 100" TO DISPLAY
    RECEIVE mark FROM KEYBOARD
END WHILE

Writing Evaluation Answers

Exam evaluation questions are typically worth 3–5 marks. To gain full marks, your answer must:

  1. Name the criterion being discussed
  2. State what it means (brief definition)
  3. Apply it to the specific program — give evidence from the code or testing
  4. Reach a justified conclusion — is the criterion met or not, and why?
Model Answer — Robustness (3 marks)

"The program is robust [1] because it validates all user inputs using WHILE loops that re-prompt the user if the entered value is out of range [1]. For example, the mark entry uses WHILE mark < 0 OR mark > 100 to reject invalid data. The program therefore handles unexpected input without crashing [1]."
