Design at Higher Level

⚠️
The Higher SDD specification requires structure diagrams and pseudocode for program design, and wireframes for user-interface design. Flowcharts are not in the Higher specification — they are a National 5 design tool.

Good design happens before coding. The design stage produces a blueprint that a programmer can follow to write the code. At Higher, the design process involves:

  1. Top-level design — a structure diagram showing the main modules of the system
  2. Stepwise refinement — breaking each module down into sub-modules until the lowest level is programmable
  3. Data flow — annotating the structure diagram to show what data moves between modules (parameters passed in and out)
  4. Pseudocode — describing the logic of each module in structured English-like notation
  5. Wireframes — sketching the layout of user interfaces

Structure Diagrams

A structure diagram (also called a structure chart or hierarchy diagram) shows the hierarchical decomposition of a software system into modules. Each box represents a module; connections show the relationship between parent and sub-modules.

Reading a Structure Diagram

  • The top-level box is the whole program or system
  • Child boxes below it are modules called by the parent
  • Data flow annotations show what parameters are passed between parent and child modules
  • Leaf nodes (lowest boxes) represent tasks that can be directly implemented

Stepwise Refinement

The process of expanding a module into its sub-steps. You repeat this process until every box at the bottom of the diagram is a single programmable step.

Example: "Find Highest Mark" module could be refined into: "Set maxIndex to 0" → "Loop through array" → "Compare and update maxIndex" → "Return result"

Data Flow

Arrows or annotations on the structure diagram show what data is passed between modules:

  • In (↓) — data passed into the subprogram (actual parameters)
  • Out (↑) — data returned from the subprogram (return value)
  • In/Out (↕) — data passed in, modified, and passed back

Pseudocode

Pseudocode is a structured, language-independent way of writing the logic of a program. The SQA Higher exam uses a specific reference language for pseudocode — answers in any pseudocode-like notation are accepted, but using the SQA style avoids ambiguity.

SQA Reference Language Conventions

Common Pseudocode Constructs
# Input / Output
RECEIVE value FROM KEYBOARD
SEND "Result: " & value TO DISPLAY

# Assignment
SET total TO total + value

# Selection
IF score >= 70 THEN
    SET grade TO "A"
ELSE IF score >= 60 THEN
    SET grade TO "B"
ELSE
    SET grade TO "C"
END IF

# Fixed loop
FOR i FROM 0 TO 9 DO
    SEND scores[i] TO DISPLAY
END FOR

# Conditional loop
WHILE score < 0 OR score > 100 DO
    RECEIVE score FROM KEYBOARD
END WHILE

# Procedure call
CALL displayResult(name, grade)

# Function definition
FUNCTION calculateGrade(mark : INTEGER) RETURNS STRING
    IF mark >= 70 THEN
        RETURN "A"
    ELSE
        RETURN "B"
    END IF
END FUNCTION
💡
Pseudocode in an exam should be precise enough to code from. Avoid vague phrases like "check the mark" — instead write the exact condition: IF mark >= 70 THEN.

Wireframes

A wireframe is a low-fidelity sketch or diagram of a user interface. It shows the layout and structure of a screen — where buttons, text boxes, labels and other elements appear — without colour, images or final styling.

Purpose of Wireframes

  • Allow the designer and client to agree on interface layout before coding begins
  • Quick and cheap to produce and modify
  • Focus attention on structure and usability rather than aesthetics
  • Can be presented to end-users to gather feedback early

What a Wireframe Includes

  • Positions of input fields, buttons, labels and output areas
  • Navigation structure (e.g. which buttons lead to which screens)
  • Annotations explaining what each element does
  • No colour or final typography — just layout
