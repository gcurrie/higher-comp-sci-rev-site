Usability Testing

Usability testing assesses how easy a website is to use for its intended audience. Real or representative users attempt to complete specific tasks on the website while an observer notes where they succeed, struggle or fail.

What Usability Testing Checks

  • Navigation — can users find their way around the site without help?
  • Clarity — is the purpose of each page immediately obvious?
  • Efficiency — can users complete tasks in a reasonable number of steps?
  • Error recovery — when users make mistakes (e.g. in forms), can they easily correct them?
  • Accessibility — can the site be used by people with visual or motor impairments?

Process

  1. Define target user profiles (personas)
  2. Write specific test cases (tasks for the user to attempt)
  3. Observe users attempting the tasks — note where they struggle
  4. Record results in a test plan
  5. Revise the website based on findings

Personas

A persona is a fictional but realistic profile of a typical user in the target audience. Personas are used to focus usability testing by ensuring you test with representative user types, not just the people who built the site.

Example Persona — Sports Club Website
Name: Margaret, 54 years old
Role: Club treasurer, long-standing member
Tech ability: Basic — uses email and Facebook, rarely shops online
Device: iPad at home, occasionally desktop PC
Goal: Check upcoming events and download the fixture list
Frustrations: Small text, too many clicks to find information, confusing menus
Example Persona — Sports Club Website
Name: Callum, 17 years old
Role: Junior member, prospective new player
Tech ability: High — daily social media and gaming; primarily mobile
Device: Smartphone (iPhone, Chrome browser)
Goal: Find out how to join the club, sign up for taster session
Frustrations: Sites that don't work on mobile, too much text, no clear "Join" button

Each persona reminds the developer that different people use the site in different ways — and that the site must cater for all of them, not just the developer's own preferences.

Test Cases (Usability)

A test case is a specific task given to a user during usability testing, along with an expected completion path and success criteria. Test cases should be realistic, representative of how real users would use the site.

Test Plan Example

Test No. Persona Task Expected outcome Actual outcome Pass?
1 Margaret Find the date of the next home fixture Navigate to Events page, read date from calendar Found in under 2 clicks ✓
2 Callum Complete the online membership form Form submitted without errors Got confused by radio button — no label visible on mobile ✗
3 Margaret Find the club's phone number Phone number visible on Contact page Found on Contact page ✓
4 Callum Navigate from Events page back to Home page Click Home in navigation bar Navigation bar not visible on mobile without scrolling ✗
💡
A failed test case is not a bad thing — it reveals a genuine usability problem that can be fixed. Tests with all passes and no fails may simply mean the tasks were too easy or not representative of real usage.

Compatibility Testing

Compatibility testing ensures the website works correctly across different browsers and devices. A site that works perfectly in Chrome may have layout or functionality issues in Firefox, Safari or Edge — or on a mobile screen.

Browsers to Test

  • Google Chrome
  • Mozilla Firefox
  • Apple Safari (especially on Mac and iPhone)
  • Microsoft Edge

Devices and Screen Sizes

  • Desktop — large screen (1280px+ wide), mouse navigation
  • Tablet — medium screen (~768px), touch navigation
  • Mobile/Smartphone — small screen (~375px), touch only

What to Check in Each Browser/Device

  • Page layout is correct — columns, navigation bar, spacing
  • All images load and display correctly
  • Navigation links all work
  • Forms submit correctly and show error messages
  • JavaScript event handlers function (onclick, hover effects)
  • Font sizes are readable without zooming
  • No horizontal scrollbar appears on mobile

Evaluation Criteria

A WDD solution is evaluated against fitness for purpose and usability.

1. Fitness for Purpose

Does the website do what it was designed to do? Compare the implemented site against the functional requirements and end-user requirements agreed at the analysis stage.

Example Evaluation — Fitness for Purpose

"The website includes all required pages (Home, Events, Contact), allows users to complete the registration form, and displays upcoming fixtures as specified in the requirements. All hyperlinks work correctly and the navigation structure matches the agreed hierarchy. The website is fit for purpose."

2. Usability

How easy is the website for the target audience to use? Evaluate against the end-user requirements, usability test results and persona goals.

Example Evaluation — Usability

"Usability testing showed that the majority of tasks were completed successfully by both personas. However, test case 2 revealed that radio button labels were not visible on mobile screens — this was corrected by adding explicit labels with for attributes. After the fix, all tasks were completed without assistance. The navigation bar is consistent across all pages and the colour contrast meets accessibility guidelines. The website is usable for the target audience."
