Analysis

Before designing or coding a website, a thorough analysis ensures that the final product meets the client's and users' needs. The analysis stage produces a clear specification to guide the rest of the project.

Purpose

The purpose of the website describes what it is trying to achieve. A purpose statement is typically one or two sentences:

Example Purpose Statement
"A website for a local sports club to allow members to view upcoming events, register for sessions and contact the club committee."

Target Audience

Identifying the target audience shapes design decisions about layout, language, colours and navigation. Consider:

  • Age range of users
  • Technical ability (novice vs experienced web users)
  • Devices typically used (desktop, mobile, tablet)
  • Accessibility needs (colour blindness, screen readers, large text)

End-User Requirements

End-user requirements are specific needs expressed by the people who will use the website. These are gathered from the client and potential users:

  • "The site must be navigable without a mouse (keyboard accessible)"
  • "All pages must load in under 3 seconds on a standard broadband connection"
  • "The site must work correctly on Chrome, Firefox and Safari"
  • "Text must be readable without zooming on a standard 13-inch screen"

Functional Requirements

Functional requirements describe what the website must do. As with SDD, these are split into inputs, processes and outputs:

CategoryWeb contextExample
Inputs Data entered by the user via forms or other interactions Name, email, session preference entered in a registration form
Processes Actions performed on input or stored data Validate form fields, filter event list by date, display image gallery
Outputs Information displayed to the user Confirmation message, filtered list of events, contact details

Website Structure

Before creating wireframes, the website structure (hierarchy) is planned. This shows all the pages in the site and how they link to each other.

Levels in a Multi-Level Website

  • Level 0 — Home page (root)
  • Level 1 — Main section pages (e.g. About, Events, Contact)
  • Level 2 — Sub-pages within sections (e.g. individual event pages)

Navigation Design

The navigation structure determines how users move between pages. At Higher level, websites must include:

  • A consistent navigation bar present on every page
  • Links that allow users to reach any page within two clicks of the home page
  • Clear visual indication of the current page
  • A breadcrumb trail or "back" links for deeper pages

Wireframes

A wireframe is a low-fidelity (rough) sketch of a web page's layout. It shows the position of elements — navigation, headings, images, text blocks, forms — without colours, fonts or final images.

What a Webpage Wireframe Shows

  • Header area (logo, site title, navigation bar)
  • Main content area (text blocks, image placeholders, forms)
  • Sidebar areas (if applicable)
  • Footer (contact info, links, copyright)
  • Annotations explaining the purpose of each element

Low-Fidelity Prototyping

A low-fidelity prototype is a simple, quick representation of the website — typically paper sketches or basic digital mockups — used to test ideas and gather feedback before full development begins.

Advantages of Low-Fidelity Prototyping

  • Very quick and cheap to produce
  • Easy to modify — changes can be made in seconds without rewriting code
  • Allows client and users to review layout and navigation before coding
  • Identifies usability issues early when they are cheapest to fix
  • Focuses feedback on structure, not cosmetic details (colours, fonts)

Limitations

  • Cannot test interactive behaviour (hover effects, form validation)
  • May not accurately represent how the final site will look on screen
  • Users may not give realistic feedback because it looks unfinished

Prototype vs Final Website

Low-Fidelity PrototypeFinal Website
ContentPlaceholder boxes and "Lorem ipsum" textReal images, text and data
ColourGreyscale / no colourFull colour scheme applied
